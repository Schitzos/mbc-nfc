# MBC Card Data, Silent Shield, and Local Ledger Specification

## 1. Authority and Scope

This document is the implementation authority for NFC card payload shape, Silent Shield integrity protection, write-counter behavior, and the local SQLite ledger boundary.

Core rule:

```text
NFC card = operational source of truth for member balance and check-in/activity state.
SQLite ledger = device-local reporting and audit store only.
```

SQLite must never override card balance, card status, active activity state, or check-in timestamp. It records transactions processed on the current app/device so Station users can count transactions, income, and audit activity processed by that device.

## 2. NFC Card Payload v1

The card payload must be compact and versioned. The canonical logical schema is JSON, but the bytes stored on the NFC card must use the Silent Shield v1 codec, not direct plain JSON. Logical JSON is used for app/domain processing and testing.

```json
{
  "v": 1,
  "cid": "MBC-000001",
  "mid": "MBR-000001",
  "name": "Rizky",
  "bal": 50000,
  "st": "A",
  "ci": {
    "a": true,
    "act": "PARKING",
    "gid": "GATE-01",
    "t": "2026-05-02T10:00:00+07:00"
  },
  "tx": [
    {
      "id": "TX-001",
      "tp": "TOPUP",
      "amt": 50000,
      "t": "2026-05-02T09:00:00+07:00",
      "dev": "STATION-01"
    }
  ],
  "ctr": 7,
  "sig": "BASE64_HMAC_SHA256"
}
```

## 2A. NFC Capacity Budget

Before real-card implementation is marked ready, the selected NFC tag/card type must be documented in `DEVICE_TEST_MATRIX.md` with writable capacity and payload size evidence.

Rules:

- The encoded protected payload must fit within the usable writable capacity of the selected NFC card/tag after NDEF/record overhead.
- The app must estimate encoded payload byte length before writing.
- If the payload does not fit, block the write with `CARD_CAPACITY_INSUFFICIENT` and keep the previous card state unchanged.
- The v1 payload should remain compact: short keys, max five logs, masked/short display references, and no unnecessary profile fields.
- If capacity is tight, optional `name` and optional `raw_card_snapshot`/debug data must be omitted before reducing required business data.

## 3. Field Definitions

| Field    | Required                                       | Description                                                                                                                                              |
| -------- | ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `v`      | Yes                                            | Schema version. Initial version is `1`.                                                                                                                  |
| `cid`    | Yes                                            | Physical/logical card ID.                                                                                                                                |
| `mid`    | Yes                                            | Internal member ID generated by the app during Station registration.                                                                                     |
| `name`   | Optional                                       | Display name. The first implementation may omit or store an empty string if member profile input is not required.                                        |
| `bal`    | Yes                                            | Current balance in rupiah. Must be an integer and must never be negative.                                                                                |
| `st`     | Yes                                            | Card status. `A` = active, `B` = blocked/reserved. MVP does not include a block/unblock flow; if a blocked card is encountered, operations are rejected. |
| `ci`     | Yes                                            | Current activity check-in state.                                                                                                                         |
| `ci.a`   | Yes                                            | Whether the member is currently checked in.                                                                                                              |
| `ci.act` | Required when `ci.a=true`                      | Activity ID/type, for example `PARKING`.                                                                                                                 |
| `ci.gid` | Required when `ci.a=true`                      | Gate/device/operator point that created the check-in.                                                                                                    |
| `ci.t`   | Required when `ci.a=true`                      | ISO-8601 check-in timestamp with timezone.                                                                                                               |
| `ci.tr`  | Required when `ci.a=true`                      | Tariff snapshot rate per started hour captured at check-in.                                                                                              |
| `ci.tv`  | Required when `ci.a=true`                      | Tariff setting version captured at check-in.                                                                                                             |
| `ci.rm`  | Required when `ci.a=true` if rounding may vary | Rounding mode captured at check-in, normally `CEIL_TO_STARTED_HOUR`.                                                                                     |
| `tx`     | Yes                                            | Latest card-side transaction logs. Maximum length is 5.                                                                                                  |
| `ctr`    | Yes                                            | Monotonic write counter incremented after every successful card-state write.                                                                             |
| `sig`    | Yes                                            | Silent Shield HMAC signature over all fields except `sig`.                                                                                               |

When `ci.a=false`, `ci.act`, `ci.gid`, `ci.t`, `ci.tr`, `ci.tv`, and `ci.rm` must be absent or empty. The app must normalize this consistently before signing.

## 4. Transaction Log Rules

Allowed card transaction types:

```text
REGISTER
TOPUP
CHECKIN
CHECKOUT
```

Card-side transaction entry shape:

```json
{
  "id": "TX-001",
  "tp": "CHECKOUT",
  "amt": 4000,
  "t": "2026-05-02T11:10:00+07:00",
  "dev": "TERMINAL-01"
}
```

Rules:

- `tx` stores only the latest five entries, newest-first.
- When appending the sixth entry, remove the oldest entry first.
- `amt` is positive for `REGISTER`, `TOPUP`, and `CHECKOUT` fee amount; `CHECKIN` may use `0`.
- Card transaction logs are a compact recent-history display, not the reporting database.

## 5. Silent Shield v1 Production-Grade Requirement

Silent Shield v1 must provide assessor-testable confidentiality and integrity for the NFC card payload. It is not enough to store plain JSON, Base64 JSON, or lightly encoded data. Generic NFC reader apps must not be able to reveal member identity, balance, check-in state, or transaction values as readable fields.

Production-grade v1 implementation model:

```text
canonicalPayloadWithoutSig = StableJson(payload excluding sig)
sig = Base64Url(HMAC-SHA256(canonicalPayloadWithoutSig, MAC_KEY))
signedPayload = StableJson(payload including sig)
protectedEnvelope = AES-256-GCM-Encrypt(signedPayload, ENC_KEY, AAD)
nfcRecord = "mbc1." + Base64Url(envelopeHeader + iv + ciphertext + authTag)
```

Envelope header must include at minimum:

| Field   | Description                                                                                          |
| ------- | ---------------------------------------------------------------------------------------------------- |
| `magic` | Fixed marker such as `MBC1` so the app can detect supported payloads without exposing business data. |
| `v`     | Shield/envelope version.                                                                             |
| `kid`   | Key identifier for future key rotation. It must not reveal the actual key.                           |
| `alg`   | Algorithm identifier, for example `A256GCM+HS256`.                                                   |

Key requirements:

- Use separate encryption and signing keys: `ENC_KEY` and `MAC_KEY`.
- Derive keys from a provisioned master secret using HKDF or equivalent key derivation when available.
- Do not commit real secrets to the repository.
- Provide `.env.example` or documented secure configuration placeholders only.
- Load assessment/demo keys from environment, secure build configuration, or platform secure storage.
- Store runtime keys using Android Keystore / iOS Keychain where platform support allows.
- Never log keys, derived keys, IVs with secrets, decrypted payloads, or raw sensitive payloads.
- Use a fresh random IV/nonce for every AES-GCM encryption operation.
- Include envelope version, key ID, and card ID reference in AES-GCM additional authenticated data where feasible.
- Use constant-time comparison for HMAC verification where available.

Read flow:

1. Read NFC record.
2. Verify supported `mbc1` envelope marker and envelope version.
3. Decode envelope.
4. Decrypt with AES-256-GCM using the key identified by `kid`.
5. If decrypt/authentication fails, reject with `CARD_TAMPERED`.
6. Parse signed logical payload.
7. Recalculate HMAC over canonical payload without `sig`.
8. If HMAC validation fails, reject with `CARD_TAMPERED`.
9. Validate schema, status, balance, check-in state, logs, timestamp, and counter before role use.

Write flow:

1. Build logical payload without `sig`.
2. Normalize empty values consistently. When `ci.a=false`, `ci.act`, `ci.gid`, and `ci.t` must be absent or empty in the same way every time.
3. Canonicalize with stable key order.
4. Sign with HMAC-SHA256 and write `sig` into the logical payload.
5. Encrypt the signed payload with AES-256-GCM.
6. Write only the protected `mbc1` envelope to NFC.
7. Read the card back, decrypt, verify HMAC, and confirm expected counter/state before showing success.

Failure message:

```text
Card data is invalid or modified. Please go to Station.
```

Hard rejection rules:

- Plain JSON on card is not allowed.
- Base64-only storage is not allowed.
- XOR/custom reversible obfuscation is not allowed.
- HMAC without encryption is not enough for this assessment because the assessor may inspect the card using generic NFC tools.
- Any invalid decrypt, invalid HMAC, malformed schema, negative balance, invalid status, invalid timestamp, or invalid check-in structure must block the role action.

## 6. Generic NFC Reader Security Acceptance

Security acceptance must be demonstrable using at least one external/generic NFC reader app or tool.

Minimum passing evidence:

- Generic NFC reader does not show plain member ID, display name, balance, check-in timestamp, or transaction amount values.
- Raw record appears as an opaque Silent Shield envelope, not readable business JSON.
- Editing one byte of the stored payload causes decrypt/authentication failure or HMAC failure.
- Copying an old locally cached card image over a newer card must be detected at least as a counter-regression warning when the device has seen the newer counter before.
- App logs and SQLite reports never expose decrypted raw payloads or encryption/signing secrets.

## 7. Write Counter and Replay Handling

`ctr` increments by `1` after each successful card-state write.

Local cache table may store last seen counter per card:

```sql
CREATE TABLE card_state_cache (
  card_id TEXT PRIMARY KEY,
  last_counter INTEGER NOT NULL,
  last_seen_at TEXT NOT NULL
);
```

Validation rule:

- If `card.ctr < local_cached_ctr`, show a warning that the card may contain outdated state.
- Do not hard block solely because of counter regression, because the system is offline and multiple devices may process different cards without shared cache.
- Always hard block invalid signature, malformed payload, negative balance, invalid status, or invalid check-in structure.

## 8. Local SQLite Ledger

SQLite is required for Station reporting and audit on the current device/current app installation only. It is not a central database and does not provide global cooperative reporting across multiple phones.

Table schema:

```sql
CREATE TABLE transactions (
  id TEXT PRIMARY KEY,
  card_id TEXT NOT NULL,
  member_id_ref TEXT NOT NULL,
  type TEXT NOT NULL,
  amount INTEGER NOT NULL,
  balance_before INTEGER,
  balance_after INTEGER,
  activity_id TEXT,
  device_role TEXT NOT NULL,
  device_id TEXT NOT NULL,
  occurred_at TEXT NOT NULL,
  raw_card_snapshot TEXT,
  created_at TEXT NOT NULL
);
```

Privacy rule:

- Prefer masked or shortened `member_id_ref` in local reporting.
- Do not store raw secrets or the raw undecoded protected NFC payload in logs.
- `raw_card_snapshot` is optional and, if used, must be redacted or protected.

Ledger write rules:

- Append a row after every successful card-state operation: `REGISTER`, `TOPUP`, `CHECKIN`, and `CHECKOUT`.
- `CHECKIN` rows use `amount = 0` and count toward activity/audit totals, not income.
- Income reports must filter/sum money-related rows, especially `TOPUP` and `CHECKOUT`.
- Station UI must label reports as local-device summaries.

Reporting examples:

```sql
SELECT type, COUNT(*) AS total_count, SUM(amount) AS total_amount
FROM transactions
GROUP BY type;
```

```sql
SELECT activity_id, COUNT(*) AS total_checkout, SUM(amount) AS income
FROM transactions
WHERE type = 'CHECKOUT'
GROUP BY activity_id;
```

## 9. Local Offline Tariff Setting

The app is offline, and tariff must still be changeable after the APK is built. Therefore, the active parking tariff is local device configuration, not backend configuration and not a hardcoded checkout constant.

Default seed value:

| Field                 | Value                  |
| --------------------- | ---------------------- |
| activity_type         | `PARKING`              |
| rate_per_started_hour | `2000`                 |
| currency              | `IDR`                  |
| rounding_mode         | `CEIL_TO_STARTED_HOUR` |
| version               | `1`                    |

Suggested SQLite table:

```sql
CREATE TABLE tariff_settings (
  id TEXT PRIMARY KEY,
  activity_type TEXT NOT NULL,
  rate_per_started_hour INTEGER NOT NULL,
  currency TEXT NOT NULL,
  rounding_mode TEXT NOT NULL,
  version INTEGER NOT NULL,
  updated_at TEXT NOT NULL,
  updated_by_role TEXT NOT NULL
);
```

Rules:

- Gate check-in must read tariff from `TariffSettingsRepository.getActiveParkingTariff()` and write a compact tariff snapshot into `ci`.
- Terminal checkout must use the tariff snapshot stored in `ci` for fee calculation.
- Terminal must show the visit tariff snapshot before deduction.
- If tariff is missing, invalid, zero, or negative, checkout must stop with `TARIFF_NOT_CONFIGURED`.
- Only authorized Station/Admin staff may update tariff; unauthorized role attempts return `TARIFF_UPDATE_UNAUTHORIZED`.
- Changing tariff on one offline device does not update other devices.
- All operational devices must be manually configured with the same active tariff before use.
- The full tariff setting is not stored on member NFC cards, but a compact active-visit tariff snapshot is stored during check-in.
- Future enhancement may support signed Tariff Config NFC cards to distribute tariff offline.

## 9A. Tariff Snapshot at Check-In

Because the app is offline and tariffs can change locally after a member has already checked in, the tariff must be locked at successful Gate check-in.

Rules:

- Gate reads the current local active parking tariff before writing check-in state.
- Gate writes a compact tariff snapshot into the NFC card active visit state.
- Terminal checkout calculates the fee from the card's active-visit tariff snapshot, not from the current local active tariff.
- Local tariff changes affect only future check-ins.
- Existing checked-in cards keep their snapshot until checkout clears the active visit state.
- The snapshot must be compact to avoid NFC capacity issues.
- Minimum snapshot fields: rate per started hour (`ci.tr`) and tariff version (`ci.tv`). Include rounding mode (`ci.rm`) when not globally fixed.
- If a legacy/demo checked-in card has no tariff snapshot, Terminal may fallback to current local tariff only after showing `TARIFF_SNAPSHOT_MISSING` warning.

Example:

| Check-in tariff | Local tariff later changed to | Checkout fee uses |
| --------------: | ----------------------------: | ----------------: |
|   Rp 2.000/hour |                 Rp 3.000/hour |     Rp 2.000/hour |
|   Rp 3.000/hour |                     No change |     Rp 3.000/hour |

## 10. Operation Ordering

### Station Registration

```text
1. Read target card if possible and reject a valid already registered MBC card with `ALREADY_REGISTERED_CARD`.
2. Build initial card payload.
3. Set balance and NOT_CHECKED_IN state.
4. Add REGISTER transaction.
5. Increment ctr.
6. Sign/protect payload with Silent Shield.
7. Write card.
8. Read back the card and verify expected counter/state/signature.
9. Insert SQLite ledger record for the successful registration.
```

### Station Top-Up

```text
1. Read and validate card.
2. Reject invalid/tampered/blocked card.
3. Add top-up amount to balance.
4. Add TOPUP transaction.
5. Increment ctr.
6. Sign/protect payload with Silent Shield.
7. Write card.
8. Read back the card and verify expected counter/state/signature.
9. Insert SQLite ledger record for the successful top-up.
```

### Gate Check-In

```text
1. Read and validate card.
2. Reject invalid/tampered/blocked card.
3. Reject if already checked in.
4. Read active parking tariff from local tariff repository.
5. Reject with `TARIFF_NOT_CONFIGURED` if tariff is missing, invalid, zero, or negative.
6. Set active activity context, check-in timestamp, and compact tariff snapshot.
7. Add CHECKIN transaction.
8. Increment ctr.
9. Sign/protect payload with Silent Shield.
10. Write card.
11. Read back the card and verify expected counter/state/signature including tariff snapshot.
12. Insert SQLite ledger record for the successful check-in with amount `0`.
```

### Terminal Check-Out

```text
1. Read and validate card.
2. Reject invalid/tampered/blocked card.
3. Reject if not checked in.
4. Read tariff snapshot from the card active visit state.
5. If snapshot is missing because of legacy/demo data, show `TARIFF_SNAPSHOT_MISSING`; fallback to current local tariff is allowed only after visible warning.
6. Display visit tariff snapshot, charged hours, and calculated fee before deduction.
7. Calculate fee using the visit tariff snapshot.
8. Reject if balance is insufficient.
9. Deduct fee.
10. Clear active check-in state.
11. Add CHECKOUT transaction.
12. Increment ctr.
13. Sign/protect payload with Silent Shield.
14. Write card.
15. Read back the card and verify expected counter/state/signature.
16. Insert SQLite ledger record for the successful checkout.
```

If card write succeeds but ledger write fails, the member operation remains successful because the card is the operational source of truth. The app must show or record a clear local reporting warning so the reporting gap is not hidden.

## 11. Required Tests

Add or maintain tests for:

- Valid v1 payload encode/decode.
- Invalid version rejection.
- Missing required field rejection.
- Negative balance rejection.
- Invalid check-in structure rejection.
- Transaction FIFO behavior after more than five entries.
- HMAC signature success.
- HMAC signature failure after tampering with balance, identity, check-in state, transaction log, or counter.
- Counter increments after successful writes.
- Counter regression warning behavior.
- SQLite ledger insert after successful register, top-up, check-in, and checkout.
- SQLite ledger does not override card balance/status/activity state.

Additional required validation tests:

- Registration rejects already registered valid MBC cards.
- Encoded protected payload size is checked against selected NFC tag/card capacity.
- `CARD_CAPACITY_INSUFFICIENT` blocks write and keeps prior card state.
- Every real NFC write performs readback verification and reports `WRITE_VERIFY_FAILED` if the card cannot be verified.
- SQLite ledger inserts after successful `REGISTER`, `TOPUP`, `CHECKIN`, and `CHECKOUT`.
- Station ledger summary labels totals as current-device/current-installation only.
- Checkout rejects invalid duration/time before balance deduction.
- Gate snapshots the active local tariff at check-in; Terminal calculates checkout fee from the card visit tariff snapshot.
- Authorized Station/Admin tariff update stores a new local active tariff without APK rebuild.
- Terminal displays active tariff before deduction.
- Insufficient balance recovery works: top-up while checked-in, then checkout succeeds.
- Gate simulation rejects future timestamps.
- Early card removal or failed readback returns `WRITE_VERIFY_FAILED` and does not show success.
